# Текущее состояние и следующие шаги

## Дата: 2026-03-20

## Что работает

- **Scan** — параллельный прогон 8648 стратегий TLS1.2 за ~15 минут (64 воркера)
- **Стратегии из ванилы** — корпус генерируется из реальных blockcheck2 скриптов, 100% совпадение
- **Эталонная стратегия работает в браузере** — `multisplit:blob=fake_default_tls:ip_ttl=10:pos=2:nodrop:repeats=1` открывает rutracker.org в Chrome
- **Отчёты** — `report.txt` (ранжированный) + `report_vanilla.txt` (совместимый с ванилой), с timestamp
- **--from-file** — парсит ванильный формат, можно скормить чужой отчёт

## Что сломано

### Критическое: ранжирование не отражает реальность

Scan находит ~500-1200 стратегий через curl HEAD. Но **не все из них работают в браузере**.

- Эталонная стратегия (из ванильного отчёта, подтверждена в браузере) — **работает**
- Наш топ-1 (`fake:autottl:padencap`) — **не работает** в браузере
- Наш топ-3 (`fakedsplit:tcp_seq=1000000`) — **не работает** в браузере

**Причина**: curl HEAD (`-I`) проверяет только TLS handshake. Стратегия может пройти handshake, но сломать реальную передачу данных. Ранжирование по производительности/простоте бесполезно — нужно ранжировать по **реальной работоспособности**.

### TIME_WAIT

Параллелизм через `--local-port` теряет ~30% стратегий из-за TCP TIME_WAIT. Живём с этим — пользователю нужна одна рабочая, а мы находим сотни.

## Следующие шаги

### 1. Команда `check` (приоритет: высокий)

Отдельный entry point для проверки стратегий с реальной передачей данных:

```
blockcheckw check --from-file report_vanilla.txt --domain rutracker.org
```

- Берёт стратегии из отчёта (ванильного или нашего)
- Прогоняет **последовательно**, без `--local-port` — как ванила
- Один nfqws2, один NFQUEUE, curl GET (не HEAD)
- Проверяет реальную загрузку данных (>= 1024 байт)
- Нашёл первую рабочую — вывел и остановился (или прошёл все)
- Никаких TIME_WAIT проблем

Это решает главную проблему: пользователь получает **гарантированно рабочую** стратегию.

### 2. Переделать ранжирование (после check)

После того как `check` покажет какие стратегии реально работают, можно:
- Использовать результат `check` как вход для ранжирования
- Или встроить data transfer прямо в ранжирование

### 3. Pipeline scan → check

Идеальный workflow:
1. `scan` — быстро (15 мин), находит ~500 кандидатов
2. `check` — медленно но точно, проверяет топ-N кандидатов с data transfer
3. Результат — 5-10 **гарантированно рабочих** стратегий
